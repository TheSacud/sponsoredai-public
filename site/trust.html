Trust Boundary

SAI measures the wait, not the work.

This is the public data-boundary contract for the SAI CLI, Windows/macOS desktop overlay, sponsor placements, and credit gateway. It explains what stays local, what can be sent, and how you can verify or stop sponsor surfaces.

Public trust repository

SAI keeps a public trust repository for the parts users should be able to inspect without exposing backend anti-abuse, billing, or sponsor-ranking internals.

GitHub: TheSacud/sponsoredai-public

It includes the public data-boundary docs, npm launcher surface, package manifests, and event schemas. It does not include private deployment config, fraud thresholds, sponsor accounting internals, or code that would make qualified-placement spoofing easier.

What stays local

SAI does not upload developer work context. The terminal runner watches output timing; the desktop overlay checks local window state; neither sends prompts or code to SAI.

PromptsNEVER SENT
Source codeNEVER SENT
Terminal output and logsNEVER SENT
Model responsesNEVER SENT
Full paths, repo URLs, shell historyNEVER SENT
Window titles, contents, screenshotsNEVER SENT

What can be sent

Sponsor placement events are limited to operational metadata needed to issue, cap, bill, and settle a placement.

$ sai privacy schema
{
  "surface": "desktop_overlay",
  "tool": "codex",
  "event": "qualified_5s",
  "duration_bucket": "10-30s",
  "attended_interactive": true,
  "ci": false,
  "country": "PT",
  "code_uploaded": false,
  "prompt_uploaded": false,
  "logs_uploaded": false
}

Transport identifiers can include installation hash, session ID, placement ID, campaign ID, CLI version, signature fields, and click tokens. They identify placements and prevent replay; they are not developer work.

How the overlay qualifies

The overlay can show one sponsor banner over a supported desktop app. A paid placement qualifies only after local checks pass.

  • A supported app is in the foreground.
  • The banner is actually visible on screen.
  • The banner is on the same monitor as the supported app.
  • There was recent keyboard or mouse input.
  • The banner stayed visible for at least five continuous seconds.
  • Frequency and campaign caps allow another placement.

The raw foreground app state, visibility geometry, and input timing stay on the machine. Only the allowlisted placement result reaches the backend.

Gateway and model traffic

Credits can be spent through the local OpenAI-compatible gateway. When wallet spend is used, SAI provisions or updates a provider key limit and the developer machine calls the provider directly. Prompt and response bodies are not proxied through the SAI backend for wallet spend.

If a developer explicitly configures their own upstream provider key, that local provider configuration wins.

User controls

Every sponsor surface can be stopped locally.

$ stop sponsor surfaces
sai config kill-switch on --reason "local preview"
sai config set frequency off
npm uninstall -g @sponsoredai/cli

Desktop overlays can also be scoped to a supported app.

$ choose desktop surfaces
sai overlay codex
sai overlay claude
sai overlay both

What sponsors see

Sponsors receive aggregate campaign reporting: served, rendered, qualified, clicked, spent, and remaining placement volume.

They do not receive prompts, code, terminal output, shell history, repository URLs, full file paths, model responses, screenshots, window titles, or window contents.

This is the first public trust surface before more of the project is opened. Backend anti-abuse internals, fraud thresholds, and sponsor accounting internals are not published here because exposing them would make qualified-placement spoofing easier.
